# 问题

自动驾驶的各个模块——检测、跟踪、建图、预测、规划——为什么不能各干各的？

工业界的做法是给每个任务单独训练一个模型，然后串起来。问题是：上游模块犯的错会一路滚雪球到下游。检测偏了一点，跟踪就跟丢了；跟踪丢了，预测就瞎猜；预测瞎猜，规划就撞墙。每个模块单独看指标都还行，但串起来之后整体崩盘。

另一条路是多任务学习——共享一个骨干网络，上面插不同的头。省算力，但任务之间可能互相拖后腿（负迁移），而且没有人管"这些任务到底该怎么配合才能让车开好"这个问题。

UniAD 问的是一个更根本的问题：如果最终目标是规划一条安全轨迹，那前面的感知和预测应该怎么组织、怎么排列、怎么互相传话，才能让规划做得最好？

# 翻译

## 一条流水线，一个目标

想象一个乐队。以前的做法是：鼓手、贝斯、吉他、键盘各自在隔音房里录音，最后混音师把音轨叠在一起。听起来能凑合，但没有默契——鼓手加速了贝斯不知道，吉他即兴了键盘跟不上。

UniAD 的做法是把所有人拉到同一个录音棚里，面对面演奏，而且所有人都知道最终目标是让主唱（规划器）唱得舒服。鼓手（检测跟踪）打的节奏要让贝斯（建图）好跟，贝斯的走向要让吉他（运动预测）好即兴，吉他的即兴要让键盘（占用预测）好铺底，所有人最终服务于主唱的旋律线。

这就是"以规划为导向"的意思：不是先把每个任务做到最好再拼起来，而是从一开始就按"怎么让规划最好"来设计整条流水线。

## 怎么做到的：Query 是通用语言

乐队成员之间怎么传话？UniAD 用了一个统一的机制：query（查询向量）。

整个系统有四个 Transformer 解码器模块，串成一条链：

```
摄像头图像 → BEV特征 → TrackFormer → MapFormer → MotionFormer → OccFormer → Planner
                         (跟踪)      (建图)      (运动预测)    (占用预测)   (规划)
```

每个模块输出的不是最终结果的数字，而是一组 query——一种压缩了的、可学习的"信息包"。下游模块拿到上游的 query，就知道上游"看到了什么、理解了什么"。

具体来说：
- TrackFormer 输出 agent query（每个被跟踪的物体一个）
- MapFormer 输出 map query（车道线、分隔带等道路元素）
- MotionFormer 同时接收 agent query 和 map query，预测所有物体未来怎么动
- OccFormer 把运动信息铺到整个场景的网格上，预测哪里会被占据
- Planner 拿着自车的 ego-vehicle query（一个特殊的 query，从跟踪模块开始就跟着自车走），结合占用预测，规划出安全轨迹

关键设计：有一个 ego-vehicle query 从头到尾贯穿整条流水线。它在 TrackFormer 里学会"我在哪"，在 MotionFormer 里学会"我周围的车要怎么动"，最后在 Planner 里变成"我该怎么开"。

## 效果如何

在 nuScenes 数据集上，UniAD 用一个网络同时做五件事（跟踪、建图、运动预测、占用预测、规划），每件事都达到或超过了之前专门做这件事的模型：

- 运动预测：minADE 比之前最好的纯视觉方法降低 38%
- 规划：L2 误差和碰撞率比 ST-P3 分别降低 51% 和 56%
- 规划甚至超过了一些用激光雷达的方法——而 UniAD 只用摄像头

更重要的是消融实验的发现：把预测模块（运动+占用）去掉，直接从感知跳到规划，效果大幅下降。这证明了"中间任务不是可有可无的装饰，而是规划的必要前提"。

# 核心概念

## Query-based 模块间通信

一句话：用可学习的向量代替硬编码的中间表示（如 bounding box），让模块之间传递的是"理解"而非"数字"。

类比：传统方法像是部门之间只能通过填表格沟通——检测部门填一张"物体位置表"交给跟踪部门。表格格式固定，信息必然丢失。Query 像是让两个部门的人直接坐在一起开会——信息是连续的、可调整的、可以根据下游需要自动学会该传什么。

为什么重要：这是 UniAD 能端到端训练的基础。如果模块之间传的是不可微分的离散结果（比如检测框的坐标），梯度就断了，下游的需求没法反馈给上游。Query 是连续的，梯度可以一路回传。

## Ego-vehicle Query

一句话：一个专门代表自车的 query，从感知阶段就开始积累信息，最终变成规划的输入。

类比：你在一个陌生城市开车。你不只是在"看路"——你同时在感知自己的位置、判断周围车的意图、预估它们的轨迹，这些信息在你脑子里是一体的。Ego-vehicle query 就是这个"一体化的自我感知"，它不是规划器临时拼凑的输入，而是从第一步就开始生长的。

为什么重要：没有它，规划器只能拿到"别人怎么动"的信息，不知道"我在这个场景里是什么角色"。有了它，规划器天然知道自车和环境的关系。

# 洞见

任务之间的排列顺序和信息流向，本身就是一种可以被设计和优化的架构决策——而且这个决策对最终性能的影响，可能比任何单个模块的改进都大。

# 博导审稿

选题眼光很好。这个问题在 2022 年确实是行业痛点——大家都在做端到端，但没人系统地回答"到底需要哪些中间任务、怎么串"。UniAD 第一个把这件事说清楚了，而且用实验证明了。问题是真缺口，不是人造的。

方法上，用 query 串联所有模块是自然的选择，Transformer 解码器的堆叠也是成熟技术。没有特别巧妙的新机制，但系统设计的完整度很高——五个任务、四个模块、一套统一的 query 接口，工程量大但逻辑自洽。

实验诚意很足。Table 2 的消融做得扎实：逐步加模块，每加一个都能看到对规划的贡献。Baseline 选得公道，和同期的 ST-P3、BEVerse、ViP3D 都比了。唯一的遗憾是只在 nuScenes 上做了，没有 closed-loop 评估——开环规划的指标（L2 误差）和真实驾驶安全性之间有多大 gap，论文没法回答。

写作功力上乘。Figure 1 的四种设计对比一目了然，把自己的定位讲得很清楚。论文结构清晰，该说的都说了。

判决：strong accept。这是一篇定义了新范式的系统性工作，后续大量端到端自动驾驶论文都在这个框架上做改进。

# 启发

迁移：UniAD 的核心思路——"从最终目标倒推中间任务的设计"——可以迁移到任何多阶段系统。比如做 RAG 系统时，检索、重排、生成三个阶段，如果以"生成质量"为导向来设计检索策略（而非追求检索本身的 recall），可能会得到更好的端到端效果。

混搭：query-based 的模块间通信机制，可以和 LLM 的 token 表示结合。如果把自动驾驶的各个模块换成 LLM 的不同 attention head 或 layer，用类似的"任务 query"在层间传递结构化信息，可能是 DriveVLM 类工作的一个设计方向。
